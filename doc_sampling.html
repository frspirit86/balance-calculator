Sampling 예약 기반 터치 판정 시스템

축구 게임에서 “이 액션이 성공하는가”를 원·콘 같은 도형(범위)으로 판정하지 않고, 미리 알고 있는 볼의 궤적 위에서 ‘시간 × 거리 창(window)’으로 판정하는 시스템입니다. GK 슛블록, 패스컷, 리시브, 수동 수비 등 ‘예약된 터치’가 존재하는 모든 상황에 같은 로직으로 적용됩니다. 이 문서는 시스템의 설계 의도·판정 모델·튜닝 레버·설계상 장단점을 정리합니다.

01 한 줄 요약

어떤 볼의 터치(패스·슛·드리블·리시브 등)가 예약되면 그 순간부터 볼의 경로를 미리 알 수 있다. 따라서 액션의 성공 여부는 “범위 안에 들어왔는가”가 아니라, “정해진 시간 안에 그 경로의 어느 지점으로 도달해 터치할 수 있는가”로 판정한다. 이 시간·거리 계산의 반복(sampling)이 시스템의 이름이다.

핵심 명제 성공 판정의 본질은 도형(shape)이 아니라 시간 × 거리 창(window)이다. 선수의 능력치(속도·가속도·방향전환·사거리)와 모션의 재생 시간이 이 창의 크기를 결정한다.

02 왜 도형 판정을 버렸나

대부분의 게임은 액션 성공 범위를 원·부채꼴·캡슐 같은 도형으로 정의한다. 구현이 단순하지만 축구처럼 ‘공을 향해 달려가 발/손을 뻗는’ 연속 동작에는 다음 문제가 생긴다.

  • 경계의 애매함 — 도형의 가장자리에서 “되냐/안 되냐”가 1px·1프레임 단위로 흔들려 체감이 불공정하다.
  • 속도·타이밍이 반영되지 않음 — 같은 위치라도 늦게 반응했는지, 빠른 선수인지가 결과에 자연스럽게 녹지 않는다.
  • 모션과 판정의 괴리 — 판정은 통과인데 실제 애니메이션은 아직 공에 닿지 못하는(또는 그 반대) 어긋남이 생긴다.
  • 디자이너 의도 표현의 한계 — “일찍 끊을까 / 끌고 가서 안전하게 잡을까” 같은 선택을 도형으로는 표현하기 어렵다.
그림 1. 도형은 “안/밖”만 본다. Sampling은 경로 위 각 지점을 “그 시점까지 도달해 터치할 수 있는가”로 본다.

03 핵심 아이디어

트리거가 발생하는 순간(리시브 타겟이 확정됨 / 슛으로 궤적이 결정됨 / 수동 수비 키 입력 등)부터 볼은 결정된 궤적을 따라간다. 즉 “미래의 공 위치와 그 시각”을 미리 안다. 수비/GK 입장에서는 그 궤적을 촘촘히 훑으며(sample), 각 지점마다 이 지점에 제시간에 닿을 수 있는지를 계산하면 된다.

  • 예약(reserved) 터치 → 볼 경로가 미리 결정 → 각 경로점 P는 “위치”와 “볼이 도달하는 시각 t(P)”를 가진다.
  • 수비수가 P에 이동해서 터치하는 데 걸리는 총 시간이 t(P) 이하면, 그 지점에서 액션이 성립한다.
  • 성립하는 지점이 하나라도 있으면 샘플링 성공. 어디서 실제로 터치할지는 별도의 정책으로 고른다.

04 판정 모델 · 3요소

한 경로점 P에 대한 판정은 세 가지 시간·거리 요소로 구성된다.

① 이동 시간 (Reach Time)

현재 위치에서 P까지 터치 사거리 이내로 몸을 가져가는 데 걸리는 시간. 두 항으로 나뉜다.

  • 선회 시간 = (현재 바라보는 방향 → P 방향의 각도) ÷ 방향전환 속도. 몸을 돌리는 데 드는 시간.
  • 주행 시간 = 가속도 모델로 (거리 − 사거리)만큼 이동하는 시간. 정지 → 가속 → 최고속도 구간을 반영.

② 모션 터치 시간 (Motion Touch Time)

도달했다고 바로 터치되는 게 아니다. 실제 게임은 Unreal Engine 5 애니메이션에 Touch Notify를 삽입해, 모션이 재생되며 그 Notify가 작동하는 시점에 터치가 성립한다. 즉 “모션 재생 시작 → 실제 접촉”까지 걸리는 시간이며, 어떤 모션(사거리)을 쓰느냐에 따라 다르다. 모션 재생 자체가 시간 비용이라는 점이 이 시스템의 핵심 디테일이다.

③ 터치 Distance (사거리)

모션이 커버하는 거리. 몸이 이 거리 이내로 들어오면 ‘터치 성립’으로 본다. 모션은 사거리별로 Short / Mid / Long 밴드를 가진다 — 짧은 모션은 가깝게, 긴 모션은 멀리 뻗지만 재생이 더 오래 걸린다.

설계 원칙 “도착 시점 = 모션 재생 시점” — 선수는 목표 지점에 도착한 바로 그 순간 모션을 재생하고, 터치는 그로부터 모션 터치 시간만큼 뒤에 성립한다. 그래서 모션 터치 시간이 판정에 리드타임으로 더해진다.

05 타임라인과 판정식

트리거부터 터치까지의 시간 예산은 아래 순서로 흐른다. 이 합이 볼 도달 시각을 넘지 않아야 한다.

그림 2. 반응 → 선회 → 이동 → 모션 순으로 시간이 소모되고, 터치 성립 시점이 볼 도달 시각 이내이면 성공. 단, 볼 도달 시각이 샘플링 시간(윈도우)을 넘으면 그 지점은 애초에 후보에서 제외된다.

모션 밴드 b(Short/Mid/Long)에 대해, 경로점 P에서의 성공 조건은:

# 모션 b로 P를 터치 완료하는 시각 touch_time(P, b) = T_반응 + T_선회(P) + T_이동(P, 사거리b) + T_모션(b) # 성공(그 지점에서 컷/세이브 성립) 조건 touch_time(P, b) ≤ t(P) ← 볼보다 먼저 터치 t(P) ≤ 샘플링 시간 ← 윈도우 안

여기서 T_이동(P, 사거리)는 (현재→P 거리 − 사거리)를 가속도 모델로 주행하는 시간이다. 사거리가 클수록(긴 모션) 주행 거리는 줄지만 T_모션은 커지는 상충(trade-off)이 생긴다.

06 모션 우선순위

한 지점을 여러 모션으로 잡을 수 있을 때, 선수는 Short → Mid → Long 순으로 “되는 가장 작은 모션”을 고른다. 정면의 쉬운 공에 굳이 풀 다이브(Long)를 쓰지 않고, 필요한 만큼만 뻗는다.

  • Short가 되면 Short — 가장 짧고 빠른 모션. 가까이 붙어야 하지만 재생이 빠르다.
  • 안 되면 Mid, 그것도 안 되면 Long — 멀리 뻗지만 재생이 느리다.
  • Long마저 안 되면 그 지점은 불성립. 모든 지점이 불성립이면 샘플링 실패.
흔한 함정 우선순위를 Long부터 검사하면 언제나 Long만 선택된다(가장 관대하므로 늘 통과). 반드시 작은 모션부터 검사해야 “필요한 만큼만 뻗는” 자연스러운 동작이 나온다.

07 판정 절차 (의사코드)

function sample(볼_궤적, 선수):
    후보 = []
    for P in 볼_궤적.경로점들:            # 촘촘히 훑는다(sampling)
        if P.t > 선수.샘플링_시간:  continue   # 윈도우 밖
        for b in [Short, Mid, Long]:        # 작은 모션부터
            tt = 반응 + 선회(P) + 이동(P, 사거리[b]) + 모션터치[b]
            if tt ≤ P.t:
                후보.append( (P, b) )      # 이 지점·이 모션으로 성립
                break                      # 더 큰 모션은 볼 필요 없음

    if 후보 is empty: return "실패"

    선택 = 정책(후보)   # earliest / latest / 여유최대 / 최대도달 …
    return "성공", 선택.지점, 선택.모션

정책(policy)은 “어디서 터치할지”를 디자이너가 고르는 축이다. 예: 가장 일찍(적극 차단), 가장 늦게(끌고 가 안전하게), 여유 최대(가장 안정적), 최대 도달(가장 멀리) 등. 성공 여부는 정책과 무관하고, 정책은 어느 성립 지점을 쓸지만 정한다.

08 적용 사례

“예약된 터치 + 알려진 궤적”이라는 조건만 맞으면 동일 엔진이 그대로 재사용된다.

상황예약 트리거궤적모션(예시 터치시간)
GK 슛블록슛으로 궤적 확정슛 → 골문다이브 Short/Mid/Long 0.75–0.90s
패스컷리시브 타겟 확정패서 → 리시버빠른 터치 Short/Mid/Long 0.20–0.30s
리시브패스 타겟이 됨패서 → 나트래핑 모션
수동 수비수비 키 입력드리블/패스 볼태클/차단 모션

즉 같은 판정 코드에 “궤적”과 “모션 파라미터”만 갈아끼우면 새로운 액션이 된다. GK 슛블록과 패스컷은 슛→골라인이 패스→리시버로 바뀐 것 외에 로직이 동일하다 — 이 문서의 두 시뮬레이터가 그 증거다.

09 튜닝 레버

모든 수치는 데이터 테이블(예: GlobalConstGK)로 노출되어 재컴파일 없이 밸런싱된다.

레버무엇을 바꾸나올리면
사거리 S/M/L각 모션의 터치 Distance더 멀리서도 터치 성립 → 성공 범위 ↑
모션 터치시간 S/M/L모션 재생 → Touch Notify 시간리드타임 ↑ → 더 일찍 결단해야 함(성공 ↓)
최고 속도 · 가속도주행(이동) 시간먼 지점까지 이동 가능 → 성공 ↑
방향전환 속도선회 시간등 뒤 공에도 빠르게 반응
반응 시간시작 지연모든 판정이 그만큼 늦어짐(성공 ↓)
샘플링 시간최대 윈도우더 늦은(먼) 지점까지 후보 인정 → 성공 ↑
밸런싱 관찰 선수가 이미 사거리 안에 서 있어(주행 0) 남는 변수가 선회 + 모션 + 반응뿐이면, 속도·가속도를 올려도 결과가 안 변한다. 이때 유효한 레버는 방향전환 속도 · 모션 터치시간 · 반응 시간이다 — 상황마다 지배적 항이 다르다는 점이 튜닝의 묘미다.

10 설계상의 장점

  • 결정론적·재현 가능 — 같은 입력이면 항상 같은 결과. 리플레이·검증·자동 밸런싱에 유리하다.
  • 경계의 애매함 제거 — “되냐/안 되냐”가 시간×거리로 명확히 갈려 체감이 공정하다.
  • 능력치가 직관적으로 연결 — 빠르고 반응 좋은 선수가 더 많이 끊는다. 스탯이 결과에 자연히 녹는다.
  • 모션과 판정의 일치 — 판정에 모션 재생 시간을 포함하므로, 애니메이션이 실제로 공에 닿는 순간과 판정이 어긋나지 않는다.
  • 디자이너 의도의 표현력 — “일찍 끊기 / 끌고 가기”를 정책으로 authored 할 수 있다.
  • 재사용성 — 궤적·모션만 갈아끼우면 GK·패스컷·리시브·수비로 확장된다.

11 한계와 열린 질문

  • 모션·이동의 결합 방식 — 본 모델은 “이동 → 모션”을 직렬로 더한다. 실제로는 모션 워핑(Motion Warping)으로 모션이 이동을 일부 흡수해 겹칠 수 있으며, 이 경우 능력치의 영향력이 커진다.
  • Touch Notify 위치 — 접촉이 모션의 중간에 발생한다면 유효 리드타임이 짧아진다. 시뮬레이터는 이를 조정 가능한 파라미터로 노출해 두 해석을 모두 실험할 수 있게 했다.
  • 이동 모션과 슛블록 모션의 시간 결합비(예: Move/ShootBlock 비율)는 상위 로직/코드에 존재하며, 본 문서의 모델은 이를 단순화한 근사다.
  • 수비 성공 확률 — 이 문서와 시뮬레이터는 “샘플링이 되냐/안 되냐”까지만 다룬다. 실제로는 성립 이후 스탯 기반 성공 확률·이벤트 분배가 이어진다(별도 문서).

12 라이브 데모
